2. Autentifikacija i autorizacija

2.1 Autentifikacija korisnika

  • JWT tokeni (RS256) generisani od Supabase Auth servisa
  • Token TTL: 1 sat, refresh token: 7 dana
  • Lozinke nikad nisu dostupne API serveru — čuvaju se isključivo u Supabase Auth (bcrypt hash)
  • Opciona dvofaktorska autentifikacija (2FA) putem Supabase TOTP

2.2 Autorizacija API poziva

  • Svaki API endpoint zahteva validan Bearer JWT token
  • Server validira potpis tokena koristeći Supabase JWT_SECRET
  • Korisnički ID (uid) se ekstraktuje iz tokena — nikad se ne preuzima iz query parametra
  • Role-Based Access Control (RBAC) — videti sekciju 10

4. Izolacija podataka po korisniku

Svaki red u svakoj tabeli sadrži user_id kolonu. Izolacija je implementirana na dva nivoa:

  1. Aplikacioni nivo: Svaki DB upit obavezno filtrira po user_id = trenutni_korisnik
  2. Bazni nivo: Row Level Security (RLS) politike u PostgreSQL-u — čak i direktan DB pristup ne može da vrati tuđe podatke

Multi-tenant arhitektura sa kancelarijskim firmama: podaci se dele unutar iste firme samo između aktivnih članova sa odgovarajućom ulogom.

5. Tok podataka kroz AI sisteme

5.1 Šta se šalje OpenAI-u

  • Delovi teksta dokumenata (chunked excerpts, max ~4000 tokena po pozivu)
  • Opisi predmeta i pitanja advokata
  • Nikad: JMBG, broj računa, medicinska dokumentacija u sirovom obliku

5.2 OpenAI politika zadržavanja

  • OpenAI ne koristi API podatke za trening modela (politika od marta 2023)
  • Podaci se zadržavaju max 30 dana radi detekcije zloupotrebe, zatim se brišu
  • EU Standard Contractual Clauses (SCC) su potpisani — pravna osnova za prenos u SAD

5.3 Pinecone

  • Čuvaju se isključivo matematički vektori (embeddings), ne originalni tekst
  • Namespace izolacija po korisniku — vektori jednog korisnika nisu dostupni drugom

Recovery Time Objective (RTO) i Recovery Point Objective (RPO)

  • RTO: max 4 sata (za katastrofalni pad infrastrukture)
  • RPO: max 24 sata (sa dnevnim backupima) / max 5 minuta (sa PITR)

7. Audit i logovanje

  • Svaki API poziv ka /api/predmeti, /api/klijenti, /api/billing se loguje sa: user_id, metod, putanja, HTTP status, IP adresa, vreme odgovora
  • Logovi se čuvaju 90 dana na Render.com logging infrastrukturi
  • Greške (HTTP 4xx/5xx) se dodatno markuju za reviziju
  • Sentry integracija za automatsko dojavljanje kritinih grešaka (opciono)

8. Upravljanje incidentima

  1. Detekcija: Automatski health check svakih 60 sekundi + Sentry alertovi
  2. Klasifikacija: Minor / Major / Critical — prema uticaju na korisnike
  3. Notifikacija: Korisnici se obaveštavaju putem Status Page (status.vindex.ai)
  4. Rešavanje: Rollback u prethodnu verziju (<15 minuta za code deploy)
  5. Post-mortem: Analiza uzroka i preventivnih mera za Critical incidente

Rok obaveštavanja o povredi podataka: 72 sata od saznanja (ZZPL čl. 52, GDPR čl. 33).
